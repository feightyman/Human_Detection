# 项目：危险区域人员闯入检测与报警系统

## 1. 核心技术栈
- 语言：Python 3
- UI 框架：PySide6
- 视觉与模型：OpenCV (`cv2`), Ultralytics YOLOv8, ByteTrack (多目标跟踪)
- 并发模型：`threading.Thread` 或 `QThread`, `queue.Queue`

## 2. 运行方式

```bash
# 激活虚拟环境
source .venv/bin/activate   # Unix
.venv\Scripts\activate      # Windows

# 启动主程序
python main.py
```

无构建步骤，暂无测试套件和 Lint 工具。

## 3. 架构规范（严格遵守）

本项目必须采用"生产者-消费者"模型，并严格遵守以下线程划分：

- **UI 主线程 (Main Thread)**：仅负责 PySide6 界面的渲染、用户交互（如鼠标绘制多边形）、接收信号并更新图像控件。**绝对禁止在任何子线程中直接操作 UI 控件。**
- **拉流线程 (Thread 1 / 生产者)**：专门负责从 Socket、本地摄像头或视频文件中读取帧，存入共享队列。
- **推理线程 (Thread 2 / 消费者)**：从队列获取最新帧，执行 YOLO 检测、ByteTrack 追踪、多边形碰撞检测（如 `cv2.pointPolygonTest`），并将处理后的图像和统计数据通过 **Qt Signal** 发送给主线程。

## 4. 实时性与队列策略

为了保证安防报警的实时性，必须避免"队列积压"（视频流延迟）：

- 生产者与消费者之间的共享队列必须设置极小的最大长度（例如 `maxsize=2` 或 `3`）。
- 当队列满时，生产者必须采用**丢弃旧帧 (Drop Frame)** 或 **覆盖旧帧** 的策略，确保推理线程每次拿到的都是当前最新的一帧。

## 5. 模块解耦与边缘部署预留

为了方便未来向边缘设备或嵌入式 Linux 开发板移植，代码必须模块化：

- **推理逻辑解耦**：创建一个基础的 `BaseDetector` 类，YOLOv8 的逻辑应封装在其子类（如 `YoloDetector`）中。主流程只调用基础接口，不要将 PyTorch 或 YOLO 的强依赖写死在线程流中。
- **多边形状态传递**：UI 线程记录的自定义区域（多边形坐标列表）应通过线程安全的机制传递给推理线程，或者推理线程每次处理时获取一份坐标的拷贝。

## 6. 代码风格

- 必须包含清晰的中文注释，特别是多线程通信和队列管理的部分。
- 做好异常捕获，特别是：
  - `cv2.imdecode` 解码失败时的处理
  - Socket 断开或连接异常时的重连/清理逻辑
  - 由于跳帧导致 ByteTrack 丢失目标时的容错处理
